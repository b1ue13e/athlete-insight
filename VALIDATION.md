# 外部校准与验证指南

本文档描述如何将 Athlete Insight 从"内部自洽"推进到"外部有效"。

## 核心原则

> 系统的判断，必须与真实教练/真实训练场景一致。

当前系统已完成：
- ✅ 内部结构可信（四维评分、位置感知、数据完整度）
- ✅ 可迭代（版本化、回归测试、校准层）
- ✅ 可解释（debug 日志、计算步骤）

下一阶段核心：**外部校准**

---

## 第一刀：人工金标准

### 目标
收集 20-30 份真人标注，验证系统输出与人工判断的重合度。

### 标注方式

找一位懂排球的人（教练/队长/长期打比赛者），对每场比赛标注：

```yaml
标注内容:
  整体表现等级: high | medium | low
  最大优点: "一句话描述"
  最大问题: "一句话描述"
  优先改进点: "一句话描述"
  不应出现的标签: ["VB_XXX", ...]
  标注置信度: high | medium | low
  备注: "可选"
```

### 已创建的标注

文件：`lib/gold-standard.ts`

当前已有 5 份人工标注样例：
- GS001: 高得分高失误型
- GS002: 防守积极型自由人
- GS003: 关键分掉链子型
- GS004: 全面低迷型
- GS005: 稳定高效型

### 运行验证

```typescript
import { runGoldStandardValidation, generateValidationReport } from "@/lib/gold-standard"

const { results, summary } = runGoldStandardValidation()
console.log(generateValidationReport())
```

### 一致性指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 整体评级准确率 | >= 80% | 系统评级与人工评级一致 |
| 优点识别准确率 | >= 70% | 系统指出的优点击中 |
| 问题识别准确率 | >= 70% | 系统指出的问题击中 |
| 标签合理性 | >= 90% | 没有给出人工认为不该出现的标签 |
| 平均一致度 | >= 70% | 综合一致性分数 |

## 第二刀：置信度前台化

### 已实现功能

报告页现在显示：

1. **可信度徽章** - 直观显示高/中/低
2. **数据质量提示** - 完整度、置信度、引擎版本
3. **低置信度警告** - 数据不足时显著提示

### 展示层级

```
用户视图：
├── 综合评分 76
├── 可信度 中 (65)
├── 一句话结论
├── 四维评分
├── 优势/问题/建议
└── 可信性说明（底部）

开发/诊断视图（_debug）：
├── 引擎版本
├── 数据质量详情
├── 计算步骤
└── 中间分数
```

### 低置信度时的提示

```
⚠️ 数据完整度不足
当前报告可信度较低，部分判断可能不够准确。
建议补充更多比赛数据后重新分析。
```

## 第三刀：回归测试 CI 化

### 自动触发条件

修改以下文件时自动运行：
- `lib/scoring-engine.ts` - 评分引擎
- `lib/scoring-version.ts` - 版本系统
- `lib/evaluation-benchmark.ts` - 评估基准
- `lib/mock-analysis.ts` - 分析逻辑
- `lib/regression-test.ts` - 回归测试
- `lib/gold-standard.ts` - 金标准
- `types/index.ts` - 类型定义

### 测试内容

1. **评估基准测试** (10份典型样例)
   - 分数区间验证
   - 标签判定验证
   - 合理性检查

2. **位置公平性测试**
   - 防守型数据公平性
   - 进攻型数据公平性

3. **分数漂移检测**
   - 与历史基准对比
   - 漂移阈值：5分

4. **人工金标准验证**
   - 与真人标注对比
   - 一致度阈值：60%

### CI 配置

文件：`.github/workflows/regression-test.yml`

```yaml
on:
  push:
    branches: [main, develop]
    paths: [评分引擎相关文件]
  pull_request:
    branches: [main, develop]
    paths: [评分引擎相关文件]
```

### 失败阻断

任何一项测试失败，PR 无法合并：
- 基准通过率 < 80%
- 分数漂移 > 5分
- 公平性测试失败
- 金标准一致度 < 60%

## 闭环实验计划

### 实验设计

**目标**：验证系统结论与真人判断的重合度

**样本**：10-20 场真实排球数据

**流程**：
1. 收集真实比赛数据
2. 真人先给判断（金标准标注）
3. 系统跑报告
4. 对比四件事：
   - 系统总评和真人是否同向
   - 系统指出的主要问题是否击中
   - 建议是否具有可执行性
   - 低置信度时系统是否真的更保守

### 成功标准

| 检查项 | 通过标准 |
|--------|----------|
| 总评同向 | >= 80% 一致 |
| 问题击中 | >= 70% 击中 |
| 建议可执行 | >= 70% 被认可 |
| 低置信度保守 | 100% 给出警告 |

## 风险控制

### 风险1：沉迷调公式

**症状**：不断微调权重，追求更顺眼的分数

**对策**：
- 定死规则：每次调权重必须新增人工标注验证
- 关注指标从"分数好看"转向"与人工一致"
- 定期回归测试，防止过度拟合

### 风险2：debug 太强，前台太复杂

**症状**：用户被技术细节淹没

**对策**：
- 严格区分用户视图和开发视图
- `_debug` 字段只在开发环境显示
- 前台只保留关键可信度信息

### 风险3：样本偏差

**症状**：人工标注都来自同一类人（如都是教练）

**对策**：
- 多样化标注者：教练、队长、球员、数据分析师
- 记录标注者角色，分析不同视角差异
- 建立共识机制（多人标注取交集）
