# 离线数据安全感 - 已实现

## 核心目标

> 比赛刚结束，用户掏出手机录一半，走到宿舍断网了，再打开，数据还在。

只要这一点成立，用户才会把它当软件，不是网页。

---

## 已实现功能

### 1. IndexedDB 草稿持久化

**文件**: `lib/offline-storage.ts`

**特性**:
- 每一步录入自动落盘到 IndexedDB
- 支持断网、刷新、关闭浏览器后恢复
- 24小时内草稿自动识别
- 进入页面优先恢复最近一次未提交草稿

**数据流程**:
```
用户输入 → 状态变化 → 自动保存到 IndexedDB → 显示"已保存"
                ↓
        断网/刷新/关闭浏览器
                ↓
        重新打开 → 检测草稿 → 弹窗询问"继续/放弃"
```

---

### 2. 同步队列

**离线提交处理**:
1. 用户提交报告
2. 先保存到 IndexedDB（pending 状态）
3. 尝试同步到 localStorage（模拟云端）
4. 网络恢复后自动重试

**三种状态**:
- 💾 已保存到设备（离线）
- 🔄 同步中...（联网上传）
- ✓ 已同步到云端（完成）

---

**位置**: 右上角常驻显示

## 用户体验流程

### 场景 1：正常流程
1. 打开 `/analysis/new/mobile`
2. 选择运动员 → 填写比赛信息
3. 每一步自动保存（显示"已保存"）
4. 提交 → 已同步 → 查看报告

### 场景 2：中途断网
1. 录到第5步，网络断开
2. 继续填写完成
3. 提交 → 💾 已保存到设备
4. 关闭浏览器，去宿舍
5. 联网后自动同步

### 场景 3：意外关闭
1. 录到第4步，手机没电关机
2. 充电后重新打开
3. 弹窗："发现未提交草稿，2小时前，进度60%"
4. 点击"继续填写" → 恢复现场

### 场景 4：弱网环境
1. 体育馆信号差，但未完全断网
2. 每步仍自动保存到 IndexedDB
3. 提交时显示"已保存到设备，同步中..."
4. 信号恢复后自动完成同步

## 技术架构

```
IndexedDB (离线持久化)
├── drafts (草稿表)
│   └── 每个运动员的最新草稿
├── pending_reports (待同步报告)
│   └── pending / syncing / synced / failed
├── sync_queue (同步队列)
│   └── 按优先级排序的同步任务
└── settings (设置)

自动保存策略:
- 每步数据变化 → 立即保存
- 显示"已保存"状态 300ms
- IndexedDB 操作异步，不阻塞 UI

同步策略:
- 网络恢复时自动触发
- 页面可见性变化时尝试同步
- 失败自动重试（最多3次）
```
