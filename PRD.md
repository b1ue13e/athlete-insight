# Athlete Insight - Product Requirements Document

## 1. 产品名称

**Athlete Insight**  
副标题：AI 运动表现分析助手

## 2. 产品定位

Athlete Insight 是一个面向运动员、教练和训练者的 Web 应用。

用户输入比赛或训练数据后，系统自动输出结构化表现分析报告，包括：
- 综合评分
- 优势与问题
- 失误归因
- 趋势对比
- 下次训练建议

### MVP 阶段优先支持
- 排球单场比赛分析
- 跑步单次训练分析
- 健身单次训练分析

**首发真正打透的是：排球单场分析**

## 3. 用户画像

### 核心用户
学生运动员 / 校队成员 / 业余爱好者
- 他们有训练数据，但不会系统分析

### 次级用户
教练 / 队长 / 训练组织者
- 他们需要快速看出谁稳定、谁失误多、谁该重点训练

### 潜在用户
跑步和健身爱好者
- 他们有记录习惯，但缺少"解释层"

## 4. 用户核心痛点

### 痛点 1：数据有了，但没有结论
比赛结束后，用户知道自己"打得一般"，但不知道具体差在哪。

### 痛点 2：不同项目的数据分散
排球看技术统计，跑步看运动软件，健身看训练日志，没有统一分析入口。

### 痛点 3：不会从数据里提训练方向
用户能看到"发球失误 5 个"，但不知道这意味着什么，也不知道下次该怎么练。

### 痛点 4：无法长期追踪成长
单次表现容易忘，趋势变化看不清。

## 5. 产品目标

### MVP 目标
让用户在 3 分钟内完成一次数据录入，并拿到一份有结论、有归因、有建议的报告。

### 核心成功标准
- 用户能完成一次分析
- 用户觉得报告"说到了点上"
- 用户愿意保存/分享报告
- 用户愿意回来查看历史对比

## 7. 用户流程

### 路径 A：首次用户
首页 → 注册/登录 → 创建运动员档案 → 新建分析 → 选择项目 → 填写数据 → 生成报告 → 保存结果

### 路径 B：回访用户
首页 → 查看最近记录 → 点进历史报告 → 看趋势 → 发起下一次分析

### 路径 C：教练/队长视角（MVP 先弱化）
后期扩展成：队伍管理 → 成员表现对比 → 训练建议分组

## 8. 功能模块

### 8.1 首页
目标：让用户立刻理解产品价值，并快速开始一次分析。

功能：
- 产品介绍
- 最近分析记录
- 快速开始按钮
- 支持查看示例报告

### 8.2 新建分析页
目标：让用户按运动项目录入数据。

支持项目：
- 排球
- 跑步
- 健身

功能：
- 选择项目
- 表单录入
- 保存草稿
- 提交分析
- 预留文件上传入口

### 8.3 报告页
目标：让用户在 30 秒内知道结论。

固定模块：
- 一句话总结
- 综合评分
- 优势
- 问题
- 原因分析
- 建议
- 关键图表
- 原始输入摘要

### 8.4 历史页
目标：沉淀复盘价值。

功能：
- 时间线展示
- 按项目筛选
- 按评分排序
- 查看单次详情
- 与历史均值对比

### 8.5 运动员档案页
目标：形成长期画像。

功能：
- 基本资料
- 常用项目
- 历史分析数量
- 长期均分
- 问题标签画像
- 趋势图

## 9. 关键逻辑

### 9.1 分析链路
```
输入数据
→ 项目模板校验
→ 指标派生计算
→ 规则引擎打分
→ 生成结构化报告 JSON
→ 前端渲染
→ 预留 Kimi 文本增强
```

### 9.2 分析原则
模型只负责"解释"，不负责"瞎算"。

也就是：
- 评分、比率、波动、风险：规则和公式算
- 文本描述、建议措辞、总结表达：LLM 生成

这是产品可信度的底线。

## 10. 排球 MVP 的核心分析项

### 输入字段
- 比赛名称
- 日期
- 位置（主攻/副攻/二传/接应/自由人）
- 总得分
- 总丢分
- 发球得分
- 发球失误
- 扣球成功
- 扣球失误
- 被拦次数
- 一传到位率
- 拦网得分
- 防守救球
- 关键分表现评分
- 失误原因标签
- 主观备注

### 派生指标
- 发球稳定度
- 进攻效率
- 失误密度
- 得失分贡献比
- 关键分稳定度
- 综合评分

### 输出重点
- 你是"高上限但不稳定"还是"稳定但缺乏终结"还是"整体低迷，短板明显"
- 这类结论要一眼看出来

## 11. 跑步和健身的兼容策略

别把表结构写死成排球专用，所以顶层要统一：
- `analysis_sessions`
- `raw_input`
- `derived_metrics`
- `report_json`

项目差异放进 JSON 字段和项目子表里。

这样后面扩展最省事。

## 12. 非功能要求

- 页面简洁、偏专业运动分析风格
- 移动端可看，但优先桌面端
- 响应快
- 表单保存稳定
- 报告渲染清晰
- 数据结构可扩展
